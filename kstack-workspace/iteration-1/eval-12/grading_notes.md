## What the skill missed or could improve

### The mandatory Theme Check-In (Step 1.5) was procedurally noted but not genuinely executed
The skill correctly included the check-in as specified — presenting themes and asking whether they match expectations. But in a document output, this became a bracketed note ("In a live session, I'd wait for your response here") that implicitly skips the gate. The skill definition says "wait for response and adjust themes before proceeding." In document/async output format, the skill needs a better mechanism: either present themes only and explicitly invite response before continuing, or produce the full synthesis with a clear header marking the check-in as bypassed and themes as provisional. The current approach is a cosmetic compliance with a substantive instruction.

### Theme 4 (AI Expectations Meet Analogue Operations) overlaps significantly with Theme 1 (The Invisible Wait)
Both themes use the same evidence (Journey Map status communication gap, Trend Scan AI expectation gap) and point to the same underlying problem (expectation-reality gap in responsiveness). Combining them into a single, sharper theme would have produced a tighter synthesis. The overlap weakens both themes and may cause confusion in an ideation session where facilitators treat them as distinct problem spaces.

### The HMW table is well-structured but some questions are still too broad
"HMW redesign the document submission and assessment stages so that processing capacity scales with application volume" is more of a design brief than an HMW question — it already specifies the solution space (scaling capacity) rather than opening it up. A better version: "HMW give Credit Operations the information they need instantly at the point of document receipt, so they can process without re-work?" The skill definition's guidance on HMW specificity ("specific enough to generate concrete ideas, open enough to allow creative responses") was partially met but not consistently.

### No explicit connection to the ideation session purpose
The user said this synthesis is "to brief our design team before an ideation session next week." The skill delivered a full synthesis but didn't tailor the output to that specific purpose — for example, by flagging which HMW questions are strongest as ideation prompts, by suggesting how to use the Burning Platform to open the session, or by noting that the Tensions section can be used as the constraint frame for creative exercises. The skill produces the same output regardless of stated purpose, which undermines the Step 0 instruction to ask what the synthesis will be used for.

### Burning Platform item 1 mixes metrics without distinguishing time dimension clearly
"Every point of NPS decline at current churn levels is costing more to recover than to have prevented" is a strong observation but the time dimension — "12–18 months at current trajectory" — is asserted without any calculation or evidence basis. The skill should either provide the reasoning (e.g., NPS trend rate, acquisition cost assumptions) or flag explicitly that this is a modelled estimate requiring validation. Presenting it as a stated fact rather than an analytical inference risks the synthesis being challenged on that number in a leadership briefing.

### Prompt 2 doesn't explain what makes a good vs. poor input
The intake response lists what to bring but doesn't set expectations about quality thresholds. A user might bring a single page of informal notes and expect a full synthesis. The skill would benefit from saying something like: "Synthesis works best with at least two distinct sources of evidence — customer experience data and business performance data. If you only have one, I'll tell you what's missing and how it limits the output."

## Specific weaknesses to fix in the skill definition

1. **Step 1.5 (Theme Check-In):** The step currently assumes a conversational/interactive context. Add a document-output variant: "If delivering in document format, mark themes as PROVISIONAL and include a clearly boxed invitation for the reader to respond before the synthesis is used in a workshop or presented to leadership. Do not present full synthesis as confirmed until check-in is completed."

2. **Step 1 (Affinity Clustering):** Add an instruction to check for theme overlap before finalising the cluster set. If two themes share more than two evidence points, they should be merged or one should be reframed to capture a genuinely distinct pattern.

3. **Step 3 (HMW):** Strengthen the specificity guidance. Add a test: "A good HMW should be falsifiable — there should be ideas that would answer it and ideas that wouldn't. If any idea could answer the HMW, it's too broad." Give one more example of a too-broad HMW to avoid alongside the existing good examples.

4. **Step 0 (Load Inputs — Purpose):** Add an instruction to use the stated purpose to tailor the output. If the stated purpose is a workshop brief, the output should include a "How to use this synthesis in your workshop" section identifying which sections map to which workshop activities. If it's a board paper, the Synthesis Headline and Burning Platform are the primary sections.

5. **Step 4 (Burning Platform):** Add a quality check: "If a time dimension is asserted as a specific timeframe (e.g., '12–18 months'), name the basis for that estimate or flag it explicitly as a modelled assumption requiring validation." Unsupported timeframes in leadership briefings create credibility risk.

6. **Intake handling (Step 0 guidance for Prompt 2 type scenarios):** Add explicit minimum viable input guidance: what's the minimum evidence needed to produce a credible synthesis, and what should the skill say when inputs fall below that threshold.
